Why

What is the idea behind putting a 'version' on a piece of software? To distinguish it and other versions/configurations of the same software. 

This 'distinguishment' can both have technical and a psychological aspects:

  • Psychological: to indicate that the software has improved in some significant way, or that it is production-ready, or the opposite: it is not production-ready
  • Technical: to indicate whether the software is compatible with other versions, and if compatible: to indicate the scope of changes that have been applied; also to distinguish between version of the same software and are not identical.

How

Multiple versioning schemes are used. See wikipedia for a detailed list, but common schemes include:

  • <Year>
    Examples: Windows 2000, Excel 98
  • <Major>.<Minor>
    Examples: UltraEdit 14.00, Ant 1.7
  • <Major>.<Minor>.<Patch/Update>
    Examples: Linux 2.6.0
  • <Major>.<Minor>.<Micro>.<Patch/Update>
    Examples: Google Chrome 2.0.172.39, Java 1.5.0_13
  • <Major>.<Minor><Quality-level>
    Examples: Windows 7RC1, Firefox 3.0.9-beta

My preferred versioning scheme: <major-version>.<minor-version><release-level>?

  • The major version is there to indicate incompatibilities
  • The minor version to indicate that functionality has been added or bugs have been fixed
  • The release level to distinguish between a software component that is released (without <release-level>) or still in test (-alpha, -beta, -rc)

Compatibility

Technically, the major benefits of a version number is indicating compatibility with other versions. The term 'compatibility' has however a different meaning for different types of software:

 Software Type What is 'compatibility' ?
 Library                     Compatible means that clients of the library can run unaltered when using the new version of the library. This means that the new version has not removed or altered any existing functionality. Adding new functionality is possible though. 
 Application                 Compatible means that all old functionality is still available and accessible in exactly the same way(s) it was previously. Any application outputs must still be in the same format as previously in order to not break other applications that work use these as input. 
 GUI Application    All old functionality is available without any learning curve: it can be accessed in the same ways (menus, toolbars, shortcuts) as previously.  Next to that, the application outputs (reports, log file, error messages, ...) are unchanged in as far as they relate to old functionality.
 CUI Application    Old command lines and old configuration files are still valid. Next to that, application outputs (reports, log files, error messages, ...) are unchanged as far as they relate to old functionality. 
 Web Application     All old functionality is still valid and accessible without any learning curve, Old URLs are still valid. Outputs (RSS feeds, XML reports, ...) are unchanged as far as they relate to old functionality. 

Expressing Software Dependencies

Expression dependencies between software packages/modules can be based on:

  • Expressing the requirement of one specific version of the software being available: no risk, but also no automatic bug fixing or other improvements
  • Expression the requirement of a specific version or any later, but compatible version: automatic bug fixing/improvements will be available, less different versions of software components need to be kept on the system, but there is a (small?) risk attached: as version numbering is a non-automatic process, versions that indicate full backward-compatibility might not be, or even when they are, a bug migth get introduced causing the client software to fail while using the newer version
The weak point in all versioning schemes is the fact that the version number is to be manually set: it would be much better if it could be determined in an automated way.
