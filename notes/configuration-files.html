Configuration Files

Configuration files seem to be an integral part of each software deployment. But are they truly essential and if so: what's the most generic way of implementing them.

Possible Categorizations

Apps. can get configured in different ways:

Categorization based on time:

  • At startup: typical for most apps
  • During runtime: typical only for interactive apps that config. gets changed at runtime; some apps (notably servers) also refresh their config. periodically or on request by rereading it from some non-interactive source

Categorization based on source of settings:

  • Interactive user interface (options menu, settings page, …)
  • Command line user interface: command line
  • File-based configuration (properties files, xml files, ini files, …)
  • Database-based configuration
  • Registry-based
  • Environment variables
  • A combination of the above where it is typical to have a ‘hierarchy’ of sources where some settings can be overwritten/overruled by ‘higher-level’ sources, while other settings are really the logical accumulation of the settings with the same name from all sources

Categorization based on priority:

  • Typically the settings that got applied at the latest moment in time are most important (typically interactive made settings overwrite those read from files/environment/… at startup)
  • Typically command line settings overrule any settings from other, non-interactive sources
  • If multiple config. files are allowed, they typically must be specified in an order where the last one overrides previous ones. Sometimes config. files are implicitly ordered based on their scope (indicated by a specific file-system location or naming pattern): local user-specific files overrule roaming user-specific files, which themselves overrule general roaming files, which themselves overrule machine-specific files
  • Registry and env. vars seem to typically belong to the lowest-level, being overruled by almost all other types of config. sources
  • Database-based config is not very typical but if applied, it typically sits at the same level as config. files, having priority over registry and env. vars but less priority than interactive or command-line settings
  • SO TYPICAL PRIORITY CHAIN (from most important to least): UI -> CLI -> DB -> FILES (in order) -> REGISTRY -> ENV. VARS

Categorization based on persistence:

  • Some settings that are used at runtime only apply at runtime and are not typically persisted after an application run has finished: e.g. specific GUI element coordinates, specific log file name, …

Configuration Data Types

Web forms, configuration/settings/ini files, configuration/settings dialogs/pages, command line options, ... are all examples of data input. Most of these ways to perform data input seem to use a finite set of underlying 'data types'. This 'core set of data types' is larger than the set of primitive types typically provided in programming languages or databases however.

This will be an attempt to list all the fundamental types that seem to be used:

  • Number: a decimal, bounded-range number
  • Text: free-form text without any more specific meaning
  • Date: a specific day in history
  • Weekday: one of the seven weekdays
  • Month: one of the twelve months
  • Time: a point in time in a 24h clock
  • Timespan: an amount of elapsed time
  • Frequency: a number indicating the amount of repetition of a recurring event
  • URL/URI: a valid URL or URI
  • File/Directory Path: a valid file (or directory) path
  • Mail Address: a valid email address
  • Phone Number: a valid phone number 
  • Address: a valid address, which typically consists of multiple more fundamental data parts
  • Country: one of the existing countries
  • ...
