Vision Statement

The Intro


As long as I work here, we are working with COBOL:
  1. We migrate from COBOL dialect A to dialect B (e.g. BS2000 COBOL to MFCobol, Wang COBOL to MFCobol, Enterprise COBOL to Fujitsu COBOL)
  2. We create frameworks in COBOL (e.g. Wang framework, CODASYL framework)
  3. We generate COBOL components and frameworks (e.g. Anugen IO modules, IDMS map modules, CODASYL IO modules, ...)

The Opportunity


COBOL is the most-used programming language in the world: billions of lines of code are being maintained in thousands of companies all over the world. So it is inevitable that for decades to come companies will have to deal with COBOL:
  • Some will want to migrate away completely (to Java, C# or some other mainstream language)
  • Some will want to rehost their code (Enterprise COBOL or COBOL2000 on mainframe to MF or Fujitsu Cobol on Linux or Windows)
  • Those that actively keep working with COBOL will want tools to analyze, document and refactor their codebase

The Frustration


As long as I work here, we complain about our lack of COBOL tools:
  1. For analysis: no COBOL parser that works with multiple dialects (so we fall back to doing grep's and wc's to have a rough idea)
  2. For analysis/conversion: no flexible tool for estimating the impact of certain changes (e.g. change type of DBKEY variable)
  3. For analysis/conversion: no flexible tool for seeing which programs call each other 
  4. For analysis/conversion: no flexible tool for finding programs that exhibit a certain problem (e.g. programs that use DBKEY vars that are used in MOVE statements)
  5. For conversion: no full (context) parser that can also regenerate the original layout (e.g. Bramblings is not full-context aware, the COCOs cannot regenerate the original layouts)
  6. For conversion: no parser that can accept multiple COBOL dialects (e.g. clumsy Knot-scripts for COBOL2000 and also Enterprise COBOL)
And we are not the only ones that need extended COBOL parsing/analysis tools:
  1. Swets has acquired a tool that analyzes where variables are used, which copybooks are unused, etc. ...
  2. All our customers ask how they can easily find out which programs need to be recompiled when we regenerate certain artefacts
  3. If they knew they could, all customers would also ask for an automatic analysis of 'dead code' at the start of a project: just analyzing the copybooks, variables, sections and paragraphs that don't appear to be used anywhere.

The Proposal


Let's make a set of COBOL tools that we can benefit from internally, and make money from externally. And that gives us back the opportunity to actually sell tools as products and not just as part of projects.

I can see this happening in 4 phases:


Phase 1: The Basics


We make a flexible COBOL lexer/preprocessor/parser that can parse all code that resembles COBOL: those parts that it recognizes as valid COBOL syntax are parsed completely, and the other pieces are categorized as 'UNKNOWN PIECE OF CODE'. We also enable the creation of parser plugins that get a chance to inspect those parts marked as 'UNKNOWN PIECE OF CODE': plugins for embedded SQL and CICS are obvious candidates to be developed.

The result of this phase is a basic COBOL tools SDK and associated documentation on how to use and built on these pieces.

Phase 2: Publicity and Maturity


The only way of making the basic components from phase 1 good enough so they can be sold as a package to 'conversion partners' is making them work on as many COBOL sources from as many dialects as possible. So ideally we need big amounts of COBOL code to run on. And we need publicity...

So we make our first online application, which will be highly successful because:
  • It will be completely free (of charge)
  • It will provide a complete hyperlinked COBOL grammar (which appears to be a rare good)
  • It will allow anyone to create an account, upload a zip of COBOL sources and then get back:
    • General statistics: line counts, statement counts, non-standard syntax, ...
    • A full hyperlinked view of the uploaded sources: variable definitions are linked to all use-locations, variable uses are linked to the definitions, goto and perform statements are linked to their target paragraphs, ...
  • And it will be unique in its kind (as far as I know)
All 'UNKNOWN PIECE OF CODE' parts found in uploaded sources will be automatically detected and reported, giving us the chance to improve our basic (parsing) components.

Note: For internal use, this webapp will already cover most of our needs when it comes to analyzing COBOL sources during a PMA.

Phase 3: Refactoring (Online)


The next logical (but technically also the most challenging) step is adding extended analysis and refactoring support:
  • Variable/Section/Copybook/Paragraph renaming: automatically applied in all possible locations
  • Find all locations impacted by changing the type of a variable
  • Dead code detection and removal: vars/sections/copybooks/paragraphs that are not called anywhere
  • Formatting (ie. pretty-printing) of code
All of this functionality will first be added to the Cobol Tooks SDK, so internally we can benefit from it directly and the library can be offered for pay as a product-extension to 'conversion partners'.

Next to that, the webapp of phase 2 will be extended and commercialized: the extra functionality will be made available at a price that is in proportion to the LOC being worked on. It will now be possible to not only upload code and analyze it, but also apply refactorings and custom edits, and then download the results again.

Two other ways in which we will try to make money of this:
  • We will offer to create custom analysis/refactoring functions 
  • We will license the whole webapp as a packaged product

Phase 4:  Cobol to Java !


Of course: 1-to-1 COBOL to ugly COBOL-feeling Java

But the extended COBOL analysis/refactoring support of phase 3 also let's us go further: 
  • Change COBOL variable types to native Java types when it is determined that it cannot hurt
  • Don't convert dead code
  • Detect GO TO's that just implement a while-loop
  • Detect sequences of MOVE/SET statements followed by a CALL as a singly API-call (as we introduce ourselves during our migrations)
  • Interactively ask the 'conversion specialist' on what to do when there is a choice between an ugly-but-sure and an pretty-but-limited-risk option for migrating a specific construct (and also record these choices so they only have to be made once)
  • ...
Integrating this into the webapp, in which one can link from the generated Java to the original COBOL construct and vice versa ... would definitely make for a compelling demo and sales argument !
