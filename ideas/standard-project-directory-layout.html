Standard Project Directory Layout

The goals are:
  • Consistency - a simple set of rules, applied everywhere and for each type of software project
  • Intuitiveness - how easy is it to find things
  • Conciseness - shorter names and shorter paths are better unless they become less readable

A project directory structure should hold all artifacts that belong to a project and it should distinguish between:
  1. Those artifacts which are actually being maintained (e.g. source code, build scripts, ...). These should be under version control.
  2. Those artifacts that get generated automatically (e.g. compiled objects, javadocs, ...)
There is still another category: those artifacts that are not maintained but that are necessary at build or at runtime (e.g. libraries, build tools, ...). These should typically not be included in a project's directory structure as they can be shared between multiple projects.

Artifacts of category 1 that typically belong to a software project:
  • The source code itself
  • Documentation (both developer and end-user); this typically doesn't include issues/tickets but it should it my opinion.
  • Test-data 
  • Other resources needed to build a complete package (e.g. icons, licence, ...)
  • Build scripts

Proposed directory layout


  • [project name]
    • <readme file>
    • <build script(s)>
    • <build properties>
    •  
    • [java]
    • [c]
    • [python]
    • [antlr]
    •  
    • [resources]
    • [docs]
    •  
    • [build]

For Java-only projects:

  • [project name]
    • readme.txt
    • build.xml
    • build.properties
    •  
    • [java]
      • [main]
        • **/*.java
      • [test]
        • **/*.java
    • [grammar]
      • **/*.g
    •  
    • [resources]
    • [docs]
    •  
    • [build]
      • [java]
      • [classes]
      • [jars]
      • [distro]
      • [javadocs]

For c-only projects:

  • [project name]
    • readme.txt
    • Makefile
    •  
    • [c]
      • **/*.c
      • **/*.h
    • [grammar]
      • **/*.y
      • **/*.l
    •  
    • [resources]
    • [docs]
    •  
    • [build]
      • [c]
      • [h]
      • [objects]
      • [libs]
      • [distro]
      • [doxygen-docs]
